# 团队交接说明

这份文档用于发给参与 Vela Android MVP 的三位开发者，帮助他们理解产品目标、分工边界和共享文档。

## 项目目标

第一版做 Android 原生 MVP，验证两个核心入口：

- App 内聊天式 AI 日程导入
- Android 桌面 4x2 小组件

MVP 先使用 mock 数据跑通完整体验，不一开始接真实 AI、真实天气、系统日历和后端存储。

## 三人分工

### App 前端开发

负责 Android App 内体验：

- 首页
- 聊天式导入页
- 候选日程列表
- 候选日程勾选、编辑、删除、确认导入
- 小日历标红
- 应用内日程数据
- 生成 `WidgetSnapshot`

关键要求：小日历只能根据已勾选候选日程标红。

### 后端 AI 开发

负责 AI 解析服务：

- 接收文本、图片、文档或语音转写文本
- 返回结构化 `EventCandidate[]`
- 提供解析摘要、置信度、缺失字段和来源依据
- 第一阶段先提供稳定 mock 接口

关键要求：后端只返回候选日程，不决定最终导入。

### Android 小组件开发

负责 4x2 桌面小组件：

- 今日日期
- 下一个日程
- 今日剩余日程数量
- 天气/准备提示
- AI 导入按钮
- 点击跳转 App 页面

关键要求：小组件只读取 `WidgetSnapshot`，不直接调用 AI。

## 开工顺序

建议按这个节奏并行：

1. App 前端先做 mock 聊天导入页和候选日程状态流。
2. 小组件开发先做 mock `WidgetSnapshot` 展示和 deep link 跳转。
3. 后端开发先做 `/v1/event-candidates:extract` 的固定 JSON mock。
4. 前端替换本地 mock 为后端接口。
5. 前端生成真实 `WidgetSnapshot`。
6. 小组件改为读取真实 `WidgetSnapshot`。

## 联调会议检查项

每次联调只看这些问题：

- 后端返回字段是否和契约一致。
- App 是否能完整展示后端候选日程。
- 勾选/取消勾选是否实时影响小日历。
- 导入时是否只导入已勾选候选项。
- App 是否生成了小组件需要的 `WidgetSnapshot`。
- 小组件点击 AI 导入是否能打开聊天导入页。
- 合并后的 debug APK 是否能安装到已连接 Android 真机并完整跑通导入和小组件流程。

## 暂不做

为了避免第一版散掉，以下能力暂不进入 MVP：

- 真实天气 API
- Android 系统日历写入
- 复杂重复日程
- 真实通知调度
- 多尺寸小组件
- 登录和云端同步
- 自动扫描微信、邮箱或系统文件

这些能力后续可以在核心闭环稳定后逐步加入。
